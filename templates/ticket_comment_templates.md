# チケットコメントテンプレート

各subagentがRedmineチケットコメントで使用するテンプレート集。

## 共通規約

- Markdown形式で記述
- `[subagent名]` プレフィックスを冒頭に付ける
- 簡潔かつ端的な記述（コンテクスト削減）
- 決定事項 + 意図・経緯を漏れなく記載

## tech-lead テンプレート

### レビュー承認時

```markdown
[tech-lead] レビュー承認。

### 確認結果
- 設計整合性: OK — {根拠}
- ドキュメント整合性: OK / 要更新（{対象}）
- コード品質: OK — {既存パターンとの一貫性確認}

### 設計判断メモ（次セッション向け）
- {このレビューで確認した設計方針・前提}
```

### 修正指示時

```markdown
[tech-lead] 修正指示。

### 指摘事項
- [事実/仮説] {指摘内容}
  - 根拠: {情報源}
  - 修正方針: {具体的な修正内容}

### 承認条件
- {修正後に満たすべき条件}
```

### エスカレーション時

```markdown
[tech-lead] エスカレーション。設計判断が必要。

### 状況
- {現在の状況}

### 判断が必要な理由
- {なぜcoderとtech-leadだけでは判断できないか}

### 選択肢
- **{案A}** → {pro/con}
- **{案B}** → {pro/con}
```

## tech-lead 判断フレームワーク

指摘は以下の2分類を明記すること:
- **[事実]**: コードと設計書の照合で確認済みの不整合。情報源（ファイルパス・設計書の該当箇所）を必ず明記
- **[仮説]**: 設計意図との不整合の可能性があるが未確認。確認すべき事項を明記

## auditor テンプレート

### 監査完了時

```markdown
[auditor] 監査完了。

### 監査対象
- チケット: #{issue_id}
- 監査観点: {規約適合性 / プロセス遵守 / 品質基準}

### 監査結果
- [適合] {適合事項}
- [不適合] {不適合事項} — 根拠: {規約・基準の該当箇所}
  - 是正対象: {修正すべき箇所}
  - 推奨対応: {具体的な是正方法}
- [観察事項] {改善推奨だが不適合ではない事項}

### 総合判定
- {合格 / 条件付き合格 / 不合格}
```

## pmo テンプレート

### ワークフロー提案時

```markdown
[pmo] ワークフロー提案。

### 推奨ワークフロー
1. [subagent名] → 作業内容
2. [subagent名] → 作業内容

### レビュー
- tech-lead常駐: 要/不要
- レビュー方式: peer-to-peer（coder⇄tech-lead）

### 監査タイミング
- coder返却時 / クローズ前
```
